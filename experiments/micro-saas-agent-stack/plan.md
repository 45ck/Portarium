# Plan

## Summary

Build the next experiment as a governed agent stack, not as a pile of new Portarium machines.

The primary actor is one OpenClaw agent role, for example `developer-machine` or
`growth-builder`. That actor uses local tool repos as tools:

- `content-machine`
- `demo-machine`
- `manual-qa-machine`
- `prompt-language` when longer control flow is needed

Portarium remains the governor. Cockpit remains the operator surface.

## Facts vs assumptions

### Confirmed facts

- Portarium defines a `Machine` as a value-producing artifact generator, distinct from adapters.
- The sibling repos here are already shaped as CLIs and harnesses that an agent can call directly.
- The previous Growth Studio live run already proved the governed OpenClaw approval loop.

### Working assumptions

- We do not need first-class Portarium machine registration for every sibling repo in this slice.
- Stubbed external systems are sufficient to validate the workflow safely.
- Cockpit review can stay black-box at first, with API and evidence snapshots captured by the
  runner.

## Implementation slices

### Slice 1: Toolchain preflight

Outcome:

- prove local repo paths, package metadata, fixture inputs, and runtime assumptions are stable

Artifacts:

- `results/toolchain-manifest.json`

### Slice 2: Stubbed micro-SaaS workspace

Outcome:

- create a sealed workspace with:
  - product brief
  - ICP and offer
  - fake CRM state
  - fake social targets
  - fake email queues
  - fake analytics sink

Artifacts:

- committed fixture files under `fixtures/`
- runtime workspace snapshot under `results/runtime/`

### Slice 3: Governed content production

Outcome:

- OpenClaw produces launch copy, campaign plan, email drafts, CTA variants, and publish intents
- `content-machine` is invoked for content artifact generation where useful

Acceptance focus:

- outbound intents are staged, not sent
- risky actions become approvals before state changes

### Slice 4: Governed outbound queueing

Outcome:

- fake social and fake email systems receive queued actions
- Portarium records approval and evidence transitions

Acceptance focus:

- no real outbound side effect
- queue state reflects approved vs blocked actions

### Slice 5: Independent verification

Outcome:

- `manual-qa-machine` verifies the generated funnel or landing experience when the host browser
  launcher is healthy
- if the host blocks that verifier, the failure is recorded and a Playwright fallback completes the
  black-box browser run
- QA report is captured as evidence

Acceptance focus:

- the producer does not grade itself
- screenshots, snapshots, and structured QA output are captured
- any verifier fallback must be explicit in the result bundle, not hidden

### Slice 6: Optional demo packaging

Outcome:

- `demo-machine` converts a validated path into a demo artifact

Acceptance focus:

- demo generation only happens after the flow is already validated

## Evidence requirements

Every live run of the full experiment should capture:

- OpenClaw stdout and stderr
- Portarium control-plane logs
- approvals snapshot
- evidence snapshot
- queued social and email state
- fake analytics events
- QA reports
- output artifacts
- `timeline.ndjson`

## Failure conditions

The experiment fails if any of the following occur:

- a sibling repo is treated as a Portarium machine without an explicit registration decision
- the agent can publish or send outside the stub boundary
- approvals are bypassed for risky outbound actions
- QA evidence is missing or generated by the same actor being evaluated
- reproducibility depends on untracked host state

## Definition of done for the next implementation step

The next coding slice is done when:

1. the experiment runner can materialize the stubbed workspace
2. toolchain discovery is written to `results/toolchain-manifest.json`
3. fake social, fake email, fake CRM, and fake analytics fixtures are committed
4. the plan can be followed without reinterpreting the machine/tool boundary
